DATA STRUCTURE

🔗 데이터 흐름 · 제작대 리뉴얼

기획 시 설계했던 테이블 관계를 dbdiagram.io로 다시 그려봤습니다. 마우스 휠로 확대·축소, 드래그로 이동하며 참조 관계를 따라가볼 수 있습니다.

실제 사내 기획서(DBML)를 바탕으로 테이블·컬럼·참조 관계를 그대로 재현했습니다. 실 밸런스 수치, 사내 문서 링크 등 민감한 데이터는 포함하지 않았습니다. 아래는 dbdiagram.io의 "DBML-in-Link" 뷰어를 임베드한 것으로, 별도 계정이나 서버 저장 없이 DBML 원문만으로 즉석에서 렌더링됩니다.

개요

제작대 리뉴얼(기획 요약 보기)을 설계할 때, 잠금 해제·레벨업·제작이라는 서로 다른 3가지 행동이 같은 재료 구조를 공유하도록 통합 재료 테이블 하나로 묶었습니다. 아래 다이어그램에서 각 테이블의 컬럼과 그 사이를 잇는 참조선을 직접 확대해서 살펴볼 수 있습니다. 예를 들어 DT_WorkbenchMeterial.GroupId에는 잠금 해제·레벨업·제작 3곳에서 뻗어나온 참조선이 동시에 모입니다.

데이터 관계도

↗ 새 탭에서 크게 보기 (임베드가 보이지 않으면 이 링크를 이용해주세요)

설계 포인트

  1. 통합 재료 테이블 재사용: 잠금 해제·레벨업·제작이 각자 재료 테이블을 따로 갖지 않고, DT_WorkbenchMeterial 하나를 GroupId로 묶어 공유하도록 설계했습니다. 재료 구성을 바꿀 때 한 곳만 수정하면 됩니다.
  2. 서버/클라이언트 필드 분리: 제작대·레시피 테이블 안에서도 서버 로직에 필요한 필드와 클라이언트 표시 전용 필드(아이콘, 정렬 순서 등)를 CLIENT 배지로 구분해, 한 테이블이 두 가지 책임을 함께 갖고 있음을 명시했습니다.
  3. 교차 등급 제작 지원: 레시피의 장비 재료 필드(RequipredItemDataId)가 결과 아이템과 다른 등급을 참조할 수 있게 설계해, "낮은 등급 장비를 재료로 더 높은 등급 장비를 만드는" 기획 요구를 데이터 구조 차원에서부터 지원했습니다.
  4. 이름 조회 체인: 제작대 이름이든 아이템 이름이든, 결국 DT_String 하나로 모이도록 설계해 다국어 처리를 일관되게 유지했습니다.

본 페이지는 채용 지원 목적의 비공개 포트폴리오입니다.
